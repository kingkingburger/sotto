## 1. Happy Path 시나리오

### HP-01. 앱 오픈 → 5일 메뉴 즉시 표시

**전제조건**
- LocalStorage 비어있음
- DB에 is_lunchbox_friendly = true 레시피 5개 이상 존재

**스텝**
1. localhost:3000 접속
2. 랜딩 페이지 로드 확인
3. "바로 추천받기" 버튼 클릭
4. /menu?days=5 로 이동 확인
5. 스켈레톤 UI 표시 확인 (로딩 중)
6. 메뉴 카드 5개 로드 완료 대기
7. 각 카드에 음식명, 이미지(또는 🍱 폴백), 조리시간, 칼로리, 난이도 배지 표시 확인
8. "Day 1 월요일" ~ "Day 5 금요일" 라벨 순서대로 표시 확인
9. 하단 "메뉴 확정하기 →" 버튼 표시 확인

**기대 결과**
- /menu?days=5 URL에 ids 파라미터가 자동으로 추가됨 (예: ?days=5&ids=uuid1,uuid2,...)
- 카드 5개 정확히 렌더링
- 스켈레톤이 실제 카드로 교체됨
- 에러 상태 없음

**심각도**: Critical

### HP-02. 메뉴 카드 클릭 → 레시피 상세 확인

**전제조건**
- HP-01 완료 후 메뉴 5개 표시 상태

**스텝**
1. 첫 번째 메뉴 카드 이미지 또는 음식명 클릭
2. /recipe/{id} 로 이동 확인
3. Hero 이미지 (h-64 영역) 표시 확인
4. 음식명, 난이도 배지, 조리시간 배지, 컨셉 태그 배지 표시 확인
5. 영양정보 섹션 (칼로리, 단백질, 탄수화물, 지방) 표시 확인
6. 재료 섹션 — 카테고리별 카드 표시 확인
7. 조리 순서 섹션 — 타임라인 형태 스텝 표시 확인
8. 요리 팁 섹션 표시 확인 (팁이 있는 레시피의 경우)
9. "메뉴로 돌아가기" 백버튼 클릭
10. /menu 페이지 복귀 및 기존 메뉴 카드 5개 유지 확인

**기대 결과**
- 레시피 상세 데이터 정상 로드
- 뒤로가기 시 메뉴 상태 유지 (URL ids 파라미터 보존)
- 뒤로가기 시 추가 API 호출 없이 캐시/URL 기반으로 메뉴 복원

**심각도**: Critical

- 현재 메뉴의 레시피 ID 목록 기록

### HP-05. 메뉴 확정 → 장보기 목록 확인

**전제조건**
- 메뉴 5개 표시 상태

**스텝**
1. 하단 "메뉴 확정하기 →" 버튼 클릭
2. /grocery?ids={recipeIds} 로 이동 확인
3. 로딩 스켈레톤 (GrocerySkeleton) 표시 확인
4. 장보기 목록 데이터 로드 완료 대기
5. 상단 헤더 "장보기 목록" 제목 및 "N일 도시락 재료 · 총 M개 항목" 부제 확인
6. 진행률 바 표시 확인 (0/M개 완료)
7. 카테고리별 섹션 표시 확인 (육류, 채소류, 달걀류 등)
8. 각 카테고리 섹션 클릭으로 접기/펼치기 확인
9. 재료 항목 체크박스 체크 — 해당 항목 취소선 + opacity 감소 확인
10. 진행률 바 증가 확인
11. 하단 "목록 복사하기" 버튼 클릭
12. "복사 완료!" 텍스트로 버튼 변경 확인 (2초 후 원복)

**기대 결과**
- 5개 레시피의 재료가 카테고리별로 병합되어 표시
- 동일 재료(동의어 포함)는 하나로 합산 표시
- 체크 상태가 LocalStorage에 저장됨
- 클립보드에 텍스트 복사됨

**심각도**: Critical

## 2. 엣지 케이스 시나리오

### EC-01. 재뽑기 10회 연속 빠르게 클릭

**전제조건**
- 메뉴 5개 표시 상태
- DB에 충분한 레시피 풀 존재 (10개 이상)

**스텝**
1. Day 1 카드의 🎲 버튼을 빠르게 10회 연속 클릭 (각 클릭 간격 100ms)
2. 첫 번째 클릭 시 버튼 disabled 상태 전환 확인
3. 2번째~10번째 클릭은 무시됨 확인 (disabled이므로)
4. 첫 번째 reroll 완료 후 버튼 재활성화 확인
5. 카드가 정확히 1번만 교체됨 확인
6. 콘솔 에러 없음 확인

**기대 결과**
- 중복 API 호출 없음 (disabled 덕분에 1회만 실행)
- 카드 1번만 교체
- 레이스 컨디션 없음
- 에러 상태 표시 없음

**주목할 구현 취약점**
- 현재 코드(menu/page.tsx:275)에서 `disabled={isRerolling}`으로 버튼 비활성화 처리 중
- 그러나 버튼 외부 영역(카드 자체)에 클릭 이벤트가 없으므로 추가 보호 불필요

**심각도**: High

### EC-02. 브라우저 뒤로가기 후 메뉴 상태 유지

**전제조건**
- /menu?days=5&ids=uuid1,...,uuid5 상태 (메뉴 로드 완료)

**스텝**
1. Day 3 카드 클릭 → /recipe/{id} 이동
2. 브라우저 뒤로가기 버튼 클릭 (또는 Alt+Left)
3. /menu?days=5&ids=uuid1,...,uuid5 복귀
4. 메뉴 카드 5개 동일하게 표시 확인 (새 API 호출 없이)
5. URL의 ids 파라미터 동일 여부 확인
6. Day 1~5 순서 동일 여부 확인

**기대 결과**
- ids URL 파라미터를 통해 동일 메뉴 복원 (hasFetched.current ref 로직)
- API /api/recommend 재호출되지만 recipeIds 파라미터로 동일 레시피 반환
- 메뉴 구성 변경 없음

**주목할 구현 취약점**
- menu/page.tsx:88-98에서 idsParam 있을 때 recipeIds로 API 재요청
- hasFetched.current ref가 컴포넌트 언마운트 시 리셋되므로 뒤로가기 시 실제로는 재요청 발생
- 동일 ids로 요청 시 동일 순서 보장 여부 검증 필요

**심각도**: High

### EC-03. 네트워크 끊김 상태에서 동작

**전제조건**
- localhost:3000 접속 후 메뉴 로드 전 네트워크 차단
- Playwright: `page.route('**/api/recommend', route => route.abort())`

**스텝**
1. 네트워크 완전 차단 상태에서 /menu?days=5 접속
2. 스켈레톤 표시 후 에러 상태 전환 확인
3. 에러 메시지 "추천 불러오기 실패" 표시 확인
4. "다시 시도" 버튼 표시 확인
5. 네트워크 복구 후 "다시 시도" 버튼 클릭
6. 메뉴 정상 로드 확인

**기대 결과**
- 사용자에게 에러 메시지 표시 (빈 화면 아님)
- "다시 시도" 버튼으로 복구 가능
- 재시도 시 page.reload() 호출 (menu/page.tsx:217)

**주목할 구현 취약점**
- 현재 reroll 실패는 silently ignore (menu/page.tsx:144 빈 catch)
- 네트워크 끊김 중 재뽑기 실패 시 유저 피드백 없음 — PRD Critical 항목
- 장보기 목록 API 실패 시 에러 표시되지만 retry 버튼 없음

**심각도**: Critical

### EC-04. 필터 조건에 맞는 레시피 0개 (fallback 동작)

**전제조건**
- DB에 특정 태그 조합(예: nutrition + budget + volume 동시)으로 필터링 시 0개 결과 발생 가능
- 또는 테스트 환경에서 API mock으로 0개 시뮬레이션

**스텝**
1. /menu?days=5&tags=nutrition,budget,volume 접속 (매우 좁은 필터)
2. 로딩 완료 후 상태 확인
3. fallback 배너 (AlertTriangle + 노란 배경) 표시 여부 확인
4. "일부 메뉴는 선택한 컨셉과 다를 수 있어요." 메시지 확인
5. 카드는 5개 정상 표시됨 확인 (fallback으로 전체 풀에서 채움)

**기대 결과**
- 0개 결과이더라도 앱이 빈 화면을 보여주지 않음
- fallback=true → 노란 안내 배너 표시
- 카드 5개 렌더링 (전체 풀 fallback)

**주목할 구현 취약점**
- fallback에도 레시피가 부족한 경우(DB 레시피 < 5개) → 카드 N개만 표시
- 이 경우 빈 카드나 undefined 렌더링 가능성 검증 필요

**심각도**: High
