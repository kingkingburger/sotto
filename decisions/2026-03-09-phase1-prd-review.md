# Arena 토론 결과: Sotto v2 Phase 1 PRD 보완

**일시**: 2026-03-09
**프리셋**: 커스텀 4인 (헤비유저, 신규유저, 모바일유저, UX전문가)
**라운드**: 3회 (독립의견 + 반박 2회)

---

## 추천안

**옵션 B: 접근성 + UX 안내 우선 (실용 균형안)**

Phase 1의 본래 목표인 "플로우 리디자인 + 디자인 리뉴얼"에 충실하면서, 코드 검증 결과 실제로 확인된 **저비용-고영향** 문제들을 먼저 해결한다. 접근성 3종(색상 대비, 터치 타겟, reduced-motion)은 모든 에이전트가 합의했고 구현 난이도가 낮다. 코치마크/에러 안내는 신규 유저 이탈 방지에 직결된다. 히스토리/즐겨찾기 같은 데이터 모델 변경이 필요한 기능은 Phase 2로 미뤄 Phase 1의 범위를 통제한다.

---

## 옵션 비교

### 옵션 A: 최소 수정 (접근성 3종만)
- **핵심**: 색상 대비 AA, 터치 타겟 44px, Framer Motion reduced-motion 대응 — 세 가지만 수정
- **장점**: 구현 1~2일이면 완료. Phase 1 범위 확장 없음. 법적/윤리적 접근성 리스크 즉시 해소
- **리스크**: 온보딩 없이 앱을 열면 음식 카드만 나오는 문제 미해결 — 신규 유저 첫 경험이 혼란스러움. PRD 4.2절 레시피 상세 재뽑기 미구현 상태 유지
- **적합 조건**: Phase 1 배포 일정이 극히 촉박하고, 초기 유저가 대부분 재방문 헤비유저일 때

### 옵션 B: 접근성 + UX 안내 우선 (실용 균형안) — 추천
- **핵심**: 접근성 3종 + 1회 코치마크 + 필터 태그 description 노출 + 에러/빈 상태 안내 + 레시피 상세 재뽑기 + 프로그레스 바 ARIA
- **장점**: 모든 에이전트가 합의한 항목 100% 커버. 신규/기존 유저 모두의 핵심 불만 해소. 데이터 모델 변경 불필요 — 프론트엔드만 수정. 3~5일 내 완료 가능
- **리스크**: 히스토리/즐겨찾기/인분 수 설정은 미포함 — 헤비유저의 반복 사용 피로 잔존. 오프라인/PWA 미대응 — 마트 지하 시나리오 미해결
- **적합 조건**: Phase 1을 "첫인상 완성" 릴리스로 포지셔닝하고, Phase 2에서 리텐션 기능을 추가할 때

### 옵션 C: 풀 보완 (히스토리 + 즐겨찾기 + 인분 수 포함)
- **핵심**: 옵션 B 전체 + 히스토리/즐겨찾기 + 인분 수 설정 + 메뉴 확정 단계 + SW 캐싱
- **장점**: PRD 대비 누락 사항 대부분 해소. 헤비유저 리텐션 확보. 오프라인 장보기 가능
- **리스크**: Zustand 스토어 구조 변경 + DB 스키마 확장(히스토리 테이블) 필요 — Phase 1 범위가 사실상 Phase 1.5. 예상 공수 2~3주. 인분 수 설정은 재료 합산 로직(grocery.ts) 전면 수정 필요
- **적합 조건**: Phase 1 배포 일정에 2주 이상 여유가 있고, 초기 유저 중 반복 사용자 비율이 높을 것으로 예상될 때

## 갈등 지도

| 쟁점 | 찬성 | 반대 | 결론 |
|------|------|------|------|
| **히스토리/즐겨찾기 Phase 1 필수** | 헤비유저 (반복 피로 해소 필수) | UX전문가, 신규유저 (Phase 2 가능) | **Phase 2** — 데이터 모델 변경 범위가 크고 Phase 1 목표에서 벗어남 |
| **인분 수 설정 Phase 1 필수** | 헤비유저 (2인 도시락족) | 나머지 3인 (언급 안 함) | **Phase 2** — 1인만 지지하고 구현 비용 높음 |
| **스와이프 재뽑기 vs 버튼 재뽑기** | 모바일유저 (네이티브 경험) | 헤비유저 (버튼 충분) | **버튼 크기 44px 확대 우선**, 스와이프는 Phase 2 |
| **조리 모드(Wake Lock + 단계 넘기기)** | 모바일유저 (즉시 필요) | UX전문가, 헤비유저 (Phase 2) | **Phase 2** — 3인 합의, 아키텍처 변경 규모 큼 |
| **SW 캐싱(오프라인)** | 모바일유저 (마트 지하) | UX전문가 (Phase 2) | **Phase 2** — 인프라 추가 필요, Phase 1 범위 초과 |
| **코치마크 vs 풀 온보딩** | 신규유저+UX전문가 (1회 코치마크) | 헤비유저 (불필요) | **1회 코치마크 채택** — 1회성이면 헤비유저도 수용 가능 |

## 합의 사항

모든(또는 대부분의) 에이전트가 최종 라운드에서 동의한 포인트:

1. **색상 대비 AA 기준 충족** — sotto-400(`#b9a07e`) on sotto-50(`#faf8f5`) ≈ 2.5:1, WCAG AA 4.5:1 미달. 구현 쉬움 (색상값 조정)
2. **터치 타겟 44px 확보** — 재뽑기 버튼 32px, 체크박스 20px → 44px. 구현 쉬움 (CSS 수정)
3. **Framer Motion reduced-motion 적용** — CSS만 처리 중, JS 애니메이션 미대응. `useReducedMotion()` 훅 1줄. 구현 쉬움
4. **필터 태그 description 노출** — `constants.ts`에 데이터 있으나 `filter-sheet.tsx`에서 미표시. 3인 합의
5. **에러/빈 상태 안내 문구** — 필터 결과 0건 시 가이드 없음. 3인 합의

## 토론 요약

### Phase 1: 독립 의견
| 역할 | 핵심 주장 |
|------|----------|
| 헤비유저 | 히스토리/즐겨찾기 부재로 반복 사용 피로. 레시피 상세 재뽑기 PRD 명시인데 미구현 |
| 신규유저 | 온보딩 전무, 필터 태그 설명 미노출, 난이도 필터 없음, 확정 단계 없이 장보기 직행 |
| 모바일유저 | 터치 타겟 전면 미달, 주방 조리 시나리오 부재, iOS Safe Area 미대응, 오프라인 없음 |
| UX전문가 | reduced-motion 미적용, 포커스 트랩 부재, 색상 대비 미달(2.5:1), ARIA 라벨 부족 |

### Phase 2: 주요 반박 포인트
- **라운드 1**: 헤비유저가 인분 수 설정 추가 발견. 신규유저가 풀 온보딩→1회 코치마크로 후퇴. UX전문가가 Phase 1 필수를 3개로 압축 → 합의의 중심축
- **라운드 2**: 4인 모두 "접근성 3종은 Phase 1 필수"로 수렴. 히스토리는 헤비유저만 Phase 1 고수. 코치마크는 신규유저+UX전문가 합의
